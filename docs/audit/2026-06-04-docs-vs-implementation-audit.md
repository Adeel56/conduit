# Conduit Documentation Audit — 2026-06-04

## Verdict
The docs are unusually high-quality in craft (disciplined ADRs with rejected-options + reversal sections, a real threat model, a build-vs-buy register) but the single authoritative agent-guidance file, `CLAUDE.md`, is categorically wrong about repository state: it declares "inception (docs only)... no application code, build file, or infrastructure yet" while a full Spring Boot 3.5.14 / Java 21 app with eight merged feature tickets (CON-3..CON-9, CON-12) sits in the tree. That is the biggest problem — it is the file most likely to be read first and trusted, and it is contradicted by the repo's own README, tickets, and `src/`. The drift is concentrated in status/staleness claims rather than design errors (the design docs are faithful to intent); the secondary theme is that no mechanism — DoD gate, PR template, CI check, ticket status field — exists to catch this class of drift. The audit's own false-positive rate was high but disciplined: 14 of 47 candidate findings (≈30%) were refuted, almost all because the doc correctly framed something as forward-looking/plan rather than as-built.

## How big organizations handle this

**(a) Doc layers and what level owns each.** Mature orgs structure docs by level and assign each an owner and an enforcement point. Conduit has strong product/vision (`problem-brief.md`), architecture (10 ADRs), security (dedicated `docs/security/`), process (CONTRIBUTING + DoD), and decision-register (`build-vs-buy.md`) layers — most teams lack the register and the reversal sections entirely. Concrete gaps:
- **API reference layer is entirely absent.** Org practice: the API reference is *generated, never hand-maintained* — annotate controllers, auto-generate OpenAPI 3 (springdoc for Spring Boot), publish via Swagger UI/Redoc, lint+diff-gate in CI (Stripe/Twilio/GitHub). Here real endpoints ship (`IngestController.java:64`, `EventController.java:56,79`, `MeController.java:17`, merged CON-10/11) with no OpenAPI spec and no springdoc dependency — contradicting `README.md:23` ("/docs is the point"). Owner: service team, generated in build.
- **Changelog/release-notes layer is missing** despite Conventional Commits + squash being adopted "to drive the changelog" (`CONTRIBUTING.md:43`). Org practice: release-please/semantic-release/git-cliff derive `CHANGELOG.md` and GitHub Releases automatically off the commit convention already in place.
- **Operational layer is empty but referenced as populated.** `docs/runbooks/` is empty; the change-and-reversal template (line 25) tells reverse plans to "point to the specific runbook" — a dangling pointer; SLOs are "to be set" (`problem-brief.md:49-52`). Org practice: per-service runbooks linked from alerts and PR reverse-plans, SLO/error-budget docs, incident/postmortem templates. Partly stage-gated (Operate is stage 7), but the README should not present it as populated.
- **Code-level layer is thin:** no `package-info.java` per domain package, no module READMEs, no published Javadoc, despite a clean package-per-domain layout. Org practice: `package-info.java` mirrors the data-model ADR so code and design stay co-located.

**(b) ADR lifecycle practice.** The ADR template and content are ahead of typical org practice (every ADR has Status/Date/Deciders, marks the CHOSEN option, lists rejected alternatives, and — rare — has a Reversal section). But the lifecycle is declared, not exercised:
- **No ADR has ever been anything but Accepted** (all 9 landed at Accepted; ADRs 0002-0008 in one commit). Org practice (Nygard/ThoughtWorks/Amazon 6-pager): open as **Proposed**, the PR review IS the ratification, flip to Accepted on merge — Proposed is where disagreement happens.
- **No ADR index** in `docs/adr/`. Org practice: adr-tools / Backstage TechDocs auto-generate `docs/adr/README.md` (Number | Title | Status | Date | Supersedes/Superseded-by).
- **No superseding/relationship links** — template has no Supersedes/Superseded-by/Related fields; cross-refs are ad hoc (only 0005:23, 0009:66). Org practice: ADRs are immutable — revise by writing a new one with reciprocal supersede links, never by editing an Accepted one.
- **Post-ADR decisions went uncaptured:** the tenant-isolation *enforcement* mechanism was explicitly deferred to "a follow-up ADR" (`data-model.md:21-25`) then decided implicitly in CON-9 (app-layer repo filter, no RLS backstop) — a security-load-bearing decision living only in a ticket + code. Org practice: a deferred/"decide at implementation time" decision becomes a tracked, blocking ADR item.

**(c) Drift-prevention mechanisms.** This is the root cause: the docs-as-decision-record culture is named (CLAUDE.md "the documentation IS the current deliverable") and docs live in-repo, but nothing *mechanically* catches drift.
- **The DoD's docs item is conditional and unenforced:** "Docs/ADR updated *if a decision was made*" (`CONTRIBUTING.md:60`) — but CLAUDE.md going stale was a *state change* (code shipped), not a decision, so the condition excludes exactly the drift that occurred. Org practice: a path-coupled gate — CODEOWNERS + a "touched src/ but no docs/" Danger/Action soft-fail; the DoD reads "docs reflect current state".
- **No `.github/PULL_REQUEST_TEMPLATE.md`** — the PR body is *supposed to be* the change-and-reversal template (CLAUDE.md:39-40) but must be copied by hand. Org practice: committed PR template auto-fills the DoD + reversal + a "docs/CLAUDE.md/ADR/ticket-status updated (or N/A)" checkbox on every PR.
- **No CODEOWNERS** — no doc path has an owner or forced-review trigger for high-drift files (CLAUDE.md, README, `docs/adr/`).
- **No ticket status field / PR backlink** — CON-3/9/12 merged with no Status marker; CON-10/11 are merged-but-untracked. Org practice: tickets in GitHub Issues with "Closes CON-N" so merge auto-transitions status; if file-based, YAML front-matter + CI check.
- **No docs CI** — no markdown link checker (would have flagged the runbook dangling pointer) and no docs-fact assertion. Org practice: lychee/markdown-link-check + Vale in CI; reference facts that *can* be generated (build command, status) are generated or self-verified, never typed twice — the duplication of the build command between docs and CI is exactly what let CLAUDE.md's Commands rot.
